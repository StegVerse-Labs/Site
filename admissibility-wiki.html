Why this page exists

This page exists so public visitors can understand what the Admissibility Wiki is, why it exists, and how the wiki's own proposal system is described using transition-table language.

It does not accept proposals, issue receipts, approve terminology, or activate proof authority. It points to the governed wiki surface and explains the current role split.

AI-governed proposal system

The AI-governed proposal system is the wiki process by which user-submitted, maintainer-submitted, browser-originated, LLM-assisted, and AI-entity-suggested changes enter structured review.

A proposal may request a new term, revised definition, equivalence claim, overlap claim, adjacent-term mapping, implementation mapping, external reference, counterexample, dispute, or proof-path example.

Submission creates a proposal record and a submission receipt. It does not create acceptance authority. Acceptance, rejection, deferral, escalation, refusal, or supersession requires a decision record.

Submission receipt timing

User proposal receipts should include timing for the submission process itself: when the proposal was received, when the receipt was issued, which submission-stage tasks ran, and when each task started and completed.

Timing supports reconstructability. Timing does not prove the submitted claim, accept the proposal, or replace the decision record.
