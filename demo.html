StegVerse Demo

This demo evaluates whether a proposed transition should be allowed to become real. The current state history is represented as commit_states. The requested next state is represented as proposed_state.

StegVerse checks the transition at the commit boundary. The question is not whether the action sounds useful. The question is whether the system still has enough valid basis to let that action bind.

What this demo evaluates

commit_states Prior system states, evidence states, risk measurements, governance measurements, or trust-relevant observations.
proposed_state The next state the system wants to commit. This is the transition being tested before it becomes effect-capable.
ALLOW The proposed transition is admissible under the current boundary.
DENY The proposed transition crosses the admissibility boundary and should not bind.
FAIL-CLOSED The system lacks enough valid structure, data, authority, or connectivity to act safely.

Real-world examples

The demo uses small numeric vectors so the boundary can be tested quickly in a browser. In a real system, those numbers would come from measurements, checks, receipts, or model outputs.

AI deployment approval

Can an agent deploy a production change?

commit_states might represent rollback capacity, authorization confidence, incident pressure, current system stability, and recent deployment history.

proposed_state represents the deployment state the agent wants to commit.

The demo asks whether the deployment is still admissible now, not whether it looked reasonable earlier in planning.

Payment evidence

Can a payment fact become a financial effect?

commit_states might represent payment verification, receipt basis, non-entitlement flags, governance authority, and current financial policy constraints.

proposed_state represents the requested interpretation of the payment event.

The demo asks whether payment evidence remains evidence only, or whether it is trying to create unauthorized entitlement.

Credential or access change

Can a privileged action proceed?

commit_states might represent identity confidence, current trust level, policy freshness, receipt continuity, and anomaly pressure.

proposed_state represents the access change about to be committed.

The demo asks whether enough current authority and evidence still exist to let the access change become real.

In all three examples, a transition can appear valid upstream and still become unsafe at the moment of execution. This is why the boundary is checked at commit time.

How to use this demo

  • Default Data restores the original sample dataset. It does not evaluate automatically.
  • Random Data fills the Data Capture Window with a random dataset. It does not evaluate automatically.
  • Clear removes all data so you can paste your own JSON.
  • Evaluate is the only button that commits the current data to SCW for Tier 1/Tier 2 evaluation.

Required: commit_states must be an array of numeric vectors, and proposed_state must be a numeric vector of the same length.

What this proves

This demo shows the distinction between evaluating a decision and allowing a transition to commit. A system may produce a plausible action, recommendation, payment interpretation, deployment request, or access change. StegVerse evaluates whether that proposed transition is admissible at the moment it would become effect-capable.

Tier 1 evaluates the commit boundary. Tier 2 produces a local no-side-effect receipt after ALLOW. DENY and FAIL-CLOSED are also valid evidence outcomes. The point is not to make every action succeed. The point is to prevent unsupported transitions from becoming real.
