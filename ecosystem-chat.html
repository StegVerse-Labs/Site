Boundary: Site is a public mirror and user-facing advancement surface. Fillable fields generate text only into the manifest window and receipt window. Those generated windows are previews until correctness, authority, rate limits, and allowed-task status are determined at submission time. Site must not claim shell access, credential access, receipt authority, transition authority, accreditation authority, administrative authority, or autonomous deployment authority.

Public guardrails

Ecosystem Chat is not a public terminal. It is an intent-to-governed-action interface. Public users can request advancement; restricted operators and authorized ecosystem entities determine whether a task may execute through an approved task definition and an auditable receipt path.
No raw shellMessages are routed as requests. Arbitrary bash, GitHub CLI, token inspection, workflow edits, and destructive commands are not accepted here.
Allowed tasks onlyExecutable tasks must be defined outside this page by governed standards, manifests, workflows, and reviewable scripts.
Authority checkedPublic advancement requests are separate from restricted administration such as deleting branches, rotating secrets, changing permissions, or modifying protected settings.
Receipts expectedAccepted executable transitions must return status and receipt data that can be audited, replayed, or rejected by the relevant authority layer.
Abuse boundedAny live gateway must enforce rate limits, validation, replay protection, and scope limits before it can accept public submissions.
Unfinished means restrictedPrototype, pending, or concept features are display-only until the governing task contract and authority path are complete.

MVP completion definition

  

    
Done for this page means a user can type a StegVerse ecosystem request, see a deterministic route classification, receive a bounded response scaffold, distinguish advancement requests from restricted administration, and preserve a local transcript without confusing Site with shell access or proof authority.

    
The next activation step is replacing the local response generator with a governed backend gateway that can call only approved StegVerse handlers, enforce rate limits and authority checks, reject raw command execution, and return an authority-issued receipt when appropriate.
